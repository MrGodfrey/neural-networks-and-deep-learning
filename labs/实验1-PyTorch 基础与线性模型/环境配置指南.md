# 实验 1 环境配置指南

这份讲义的目标很明确：帮助你把实验 1 的环境配好，同时把最核心的几个概念真正讲明白。学完以后，你应该能独立完成下面这些事：

1. 知道为什么环境配置要从终端讲起。
2. 知道这门课里最重要的几个工具名词到底是什么意思。
3. 知道如何创建 `nndl` 环境、安装实验 1 需要的依赖，并在 VS Code 中跑通 Notebook。
4. 能把这些工具继续用到后面的实验、科研项目，以及借助 AI 快速搭建应用的流程里。

这份讲义不会把所有电脑知识一口气讲完。我们只抓住最核心、最能立刻用起来的部分。

---

## 一、为什么环境配置要从终端讲起

很多同学第一次配环境时，习惯是照着截图一步一步点。这样做有一个明显问题：只要你的电脑界面、系统版本、软件版本和教程截图稍微不一样，就很容易卡住。更麻烦的是，卡住以后往往不知道自己到底卡在什么地方。

环境配置这件事，本质上有三件事：

1. 让电脑找到你要用的程序。
2. 让 **Python** 用到正确的环境和正确的包。
3. 让 **VS Code** 连到你刚刚配好的那套环境。

> 概念：Python 是这门课运行代码最核心的工具，你可以先把它理解成“负责执行实验代码的那套程序”。

> 概念：VS Code 是一个写代码、看文件、运行 Notebook 的编辑器，可以把它理解成程序员常用的工作台。

这三件事背后，最重要的入口就是终端。  
通俗地说，终端是为了让你更高效地操作整台电脑。

所以这份讲义会先讲终端和命令行的基本概念，再讲路径、命令、Python 环境和依赖管理，最后再落到具体安装步骤。

## 二、终端、UI 和命令行入口

### 1. 先把三个最容易混的词分清楚

先说一个你每天都在接触的概念：**UI**。

> 概念：UI 就是你看到并点击的界面，比如按钮、菜单、图标、输入框。

平时你打开微信、浏览器、设置面板，看到的基本都是 UI。更准确一点说，这类界面大多属于 **图形用户界面**，英文常写作 **GUI（Graphical User Interface）**。

接下来是这节课最重要的两个词：

- **终端（Terminal）**
- **Shell**

> 概念：终端是一个让你直接输入文字命令、查看电脑输出结果的窗口。

> 概念：Shell 是终端里真正负责“读懂命令并执行命令”的程序。

它们的关系可以这样理解：

- 终端像一个窗口，负责显示内容，接收你的键盘输入。
- Shell 像窗口后面真正做事的那一层，负责理解你输入的命令，并调用电脑里的程序去执行。

### 2. 在不同系统里，它们长什么样

在不同系统里，这两个概念经常会混在一起说，所以初学者很容易弄乱。这里把它们分清楚：

- 在 **macOS** 上，你常见到的终端程序就叫 **Terminal**。打开它以后，里面默认运行的 Shell 通常是 `zsh`。
- 在 **Windows** 上，你常见到的是 **Windows Terminal**、**PowerShell** 或 **命令提示符（cmd）**。  
  其中：
  - **Windows Terminal** 更像一个终端程序，可以装不同的 Shell。
  - **PowerShell** 和 **cmd** 更接近 Shell。
  - 如果你安装了 Miniconda，Windows 上还常会用到 **Anaconda Prompt**。它本质上也是一个已经帮你提前处理好部分环境设置的命令行窗口。
- 在 **Linux** 上，不同发行版的终端程序名字可能不同，但原理一样，常见的 Shell 是 `bash`。

这节课里，你不需要死记所有名称。你只要抓住这层关系：  
**终端是你输入命令的地方，Shell 是在终端里执行命令的程序。**

### 3. 为什么实验课一定要讲它

因为很多开发工具没有传统的按钮界面，或者它们最强大的功能只在命令行里提供。

做深度学习实验时，你会频繁做这些事情：

- 创建 Python 环境
- 安装依赖包
- 查看当前 Python 是哪一个
- 运行脚本
- 启动 Notebook
- 停止卡住的程序

这些操作用终端完成，速度更快，也更清楚。你输入了什么命令，系统返回了什么结果，都会直接显示出来。

### 4. 不用它会怎样

如果完全依赖点按钮，你很容易遇到下面这些问题：

- 一换系统界面就不知道下一步在哪里。
- 一换软件版本就找不到原来的按钮。
- 遇到报错时，很难判断是路径错了、环境错了，还是包没装好。
- 很多工具本来就没有 UI，只能在命令行里使用。

也就是说，只会点按钮，往往只能完成“别人已经帮你设计好的流程”。一旦流程稍微变化，处理起来就会很被动。

### 5. 一个贴近实际工作的例子：Pandoc

假设你有一个 Word 文档 `report.docx`，想把它转成 Markdown 文档 `report.md`。  
这时可以使用一个常见的命令行工具 **Pandoc**。

> 概念：Pandoc 是一个文档格式转换工具，可以把 Word、Markdown、HTML、PDF 等格式互相转换。

如果你已经安装了 Pandoc，在终端里一条命令就可以完成转换：

```bash
pandoc report.docx -o report.md
```

这条命令的意思是：

- 读取 `report.docx`
- 输出为 `report.md`

这个例子很重要，因为它说明了一件事：  
很多真正强大的工具没有复杂的图形界面，它们主要通过终端使用。

如果你只熟悉 UI，可能会期待“打开软件 - 点击导入 - 选择格式 - 点击导出”。  
命令行工具的思路更直接：你把目标说清楚，电脑就执行。

而且一旦你有 20 个文档要转换，终端的优势会更明显。命令行天然适合重复任务、批量任务和自动化任务。

### 6. 一个总的判断

所以，终端不只是“程序员喜欢用的东西”。  
它本质上是一种更高效地操作整台电脑的方式。

## 三、以 macOS 为例：打开 Terminal 后你会看到什么

如果你在 Mac 上打开 Terminal，常见的画面大概像这样：

```text
Last login: Tue Apr 16 09:12:31 on ttys000
wangyu@MacBook-Air ~ %
```

这两行分别表示：

- `Last login: ...`：上一次登录这个终端会话的大致信息。
- `wangyu`：当前用户名。
- `MacBook-Air`：这台电脑的名字。
- `~`：当前所在位置是“家目录”，也就是你自己的用户目录。
- `%`：提示符，表示 Shell 已经准备好，可以接收命令了。

有些同学会担心：“为什么这里什么都没有，只看到一行字？”  
其实这正是终端的特点。它不靠按钮提示你下一步做什么，它只告诉你：**电脑已经准备好了，请你发出明确的指令。**

如果你后来进入了课程目录，提示符附近的内容也会跟着变化。例如进入 `nndlClass` 目录后，很多同学会看到类似下面的样子：

```text
wangyu@MacBook-Air nndlClass %
```

这说明你现在已经站在 `nndlClass` 这个文件夹里了。

## 四、一个命令到底是怎么构成的

学终端时，最重要的不是死记命令，而是知道一条命令通常由什么组成。

### 1. 最常见的结构

一条命令最常见的结构是：

```text
命令 选项 argument
```

这里只需要先记三个词：

- **命令（command）**：你想让电脑做什么。
- **选项（option）**：你想让它用什么方式做，通常写成 `-l`、`--help` 这种形式。
- **argument**：你想让它处理哪个对象。

> 概念：argument 就是写在命令后面的补充信息，用来告诉电脑“你要处理谁”或者“目标是什么”。

### 2. 看一个最简单的例子

比如这条命令：

```bash
cd labs
```

它可以拆成两部分：

- `cd`：命令
- `labs`：argument

意思就是：进入 `labs` 这个目录。

再看一条稍微复杂一点的：

```bash
ls -l labs
```

它可以拆成三部分：

- `ls`：命令
- `-l`：选项，表示用更详细的格式显示
- `labs`：argument，表示查看 `labs` 这个目录

### 3. 为什么要懂这个结构

因为以后你会频繁看到 AI、文档、老师、同学给你这样的命令：

```bash
python train.py --epochs 10
```

如果你知道命令的结构，就不会把整行内容当成一串神秘字符。你会知道：

- `python` 是让 Python 来执行
- `train.py` 是要执行的脚本
- `--epochs 10` 是额外参数

这会显著降低你使用 AI 辅助编程时的认知负担。AI 给你一条命令，你知道它在干什么，也更容易判断它能不能安全执行。

## 五、你最先要会的几个命令：`pwd`、`ls`、`cd`

学会终端以后，第一件必须弄明白的事情就是：**你现在站在哪个文件夹里。**

### 1. `pwd`

```bash
pwd
```

`pwd` 是 **print working directory** 的缩写。

> 概念：working directory 就是“当前工作目录”，也就是你现在这个终端默认正站在哪个文件夹里。

`pwd` 的作用，就是把你当前所在的位置打印出来。

### 2. `ls`

```bash
ls
```

`ls` 是 **list** 的缩写，意思是“列出”。

它会把当前目录里的文件和文件夹列出来。

### 3. `cd`

```bash
cd labs
```

`cd` 是 **change directory** 的缩写，意思是“切换目录”。

它的作用就是从当前目录进入另一个目录。

### 4. `.` 和 `..`

这两个符号特别常用：

- `.` 表示当前目录
- `..` 表示上一级目录

例如：

```bash
code .
cd ..
```

它们分别表示：

- `code .`：用 VS Code 打开当前目录
- `cd ..`：回到上一级目录

这两个符号在科研和借助 AI 建立应用时特别常见，因为很多项目文档、很多 AI 生成的命令、很多脚本写法都会用到相对路径。

举几个很实际的场景：

1. 你在课程根目录里执行 `code .`，VS Code 就会直接打开当前项目。
2. 你在 `labs/实验1-PyTorch 基础与线性模型/` 目录里，想退回到 `labs/`，可以执行 `cd ..`。
3. 你运行 AI 给出的命令时，经常会看到 `./data`、`./app.py`、`../outputs` 这种写法：
   - `./data` 表示当前目录里的 `data`
   - `../outputs` 表示上一级目录里的 `outputs`

### 5. 一个简单练习

如果你的课程文件夹叫 `nndlClass`，你可以这样操作：

```bash
cd /path/to/nndlClass
pwd
ls
code .
```

你会看到：

- `pwd` 告诉你当前所在位置
- `ls` 列出当前目录里的文件和文件夹
- `code .` 直接把当前目录作为项目打开

这就是之后做实验最常见的一组动作。

## 六、终端为什么不可替代：它代表了什么样的思维方式

终端重要，不只是因为它“能用来装包”。它背后其实代表了一种很核心的工作方式。

### 1. 明确

在终端里，你必须把动作说清楚。  
你要进入哪个目录，运行哪个脚本，安装哪个包，都会以文字形式明确写出来。

这种明确性有一个很大的好处：  
你以后自己回头看，能知道当时到底做了什么；别人帮你排查问题时，也更容易看懂你的步骤。

### 2. 可复现

按钮点击很难完整记录，命令却很容易复制、粘贴、保存和重复执行。

比如：

```bash
conda activate nndl
python -m pip install numpy matplotlib
```

这两行命令可以：

- 写进讲义
- 发给同学
- 交给 AI 解释
- 记录进实验文档

这就是终端在科研和工程里一直不可替代的原因之一：  
**它天然适合复现。**

### 3. 可自动化

终端特别适合把很多小任务连接起来，反复执行。

科研里常见的情况是：

- 跑实验
- 保存结果
- 处理数据
- 重复做多次对比

借助 AI 快速搭建应用时也一样。很多 AI 会给你明确的命令步骤，比如：

```bash
cd my_app
python -m pip install -r requirements.txt
code .
```

如果你理解终端，这些步骤就会非常清楚。  
如果你完全不理解，你就只能“照抄”，一旦出错就不知道哪一步出了问题。

### 4. 成功时经常很安静

这是终端一个非常重要的设计习惯：  
**很多命令成功时不会专门说“成功了”。**

例如：

```bash
cd labs
```

这条命令如果成功，往往不会输出任何内容。  
你可能只会看到提示符变化了，表示你已经进入了 `labs`。

这不是终端“没反应”，而是一种常见设计理念：  
**该命令的目标已经完成，就不额外打扰你。**

这样做的好处是，屏幕不会被大量无意义的“success”刷满。真正重要的信息，比如报错、警告、结果输出，会更显眼。

### 5. 它为什么适合和 AI 配合

终端是文字，AI 也擅长处理文字。  
这意味着：

- AI 可以生成命令
- 你可以让 AI 解释命令
- 你可以把报错原文贴给 AI 分析
- 你可以把一整套命令步骤保存成自己的标准流程

所以终端代表的不是“更难的操作方式”，而是更清晰、更可复现、更适合与 AI 协作的工作方式。

## 七、路径、当前目录和 `PATH`

### 1. 先理解路径

电脑里的文件和文件夹有层级结构。  
你可以把它理解成一棵树：

- 最外层有很多大文件夹
- 每个大文件夹里面又有小文件夹
- 小文件夹里面再放文件

**路径（Path）** 就是“某个文件或文件夹在这棵树上的位置”。

### 2. 当前目录为什么重要

程序通常会从“当前目录”出发找文件。

比如你想打开课程里的某个 Notebook，或者运行某个脚本。如果你当前目录不对，程序就可能找不到对应的文件。

很多新手看到“找不到文件”时，会以为是文件丢了。实际情况往往更简单：文件还在，只是你当前站错地方了。

### 3. 当你输入 `python` 时，电脑到底在找什么

当你在终端里输入：

```bash
python
```

电脑并不会把整块硬盘从头翻到尾。它会去检查一串预先约定好的目录，看看这些目录里有没有叫 `python` 的程序。  
这一串“按顺序查找程序的目录列表”，就叫 **`PATH` 环境变量**。

> 概念：`PATH` 可以理解成电脑查找程序时使用的一张名单，名单靠前的位置会被优先检查。

### 4. 为什么要懂 `PATH`

因为你后面会同时遇到很多名字相同的程序，比如：

- 系统自带的 Python
- 你自己安装的 Python
- `conda` 环境里的 Python

它们都可能叫 `python`，但实际位置不同，功能也可能不同。

如果你不理解 `PATH`，就很难搞清楚现在终端里调用的到底是哪一个 Python。

### 5. 一个特别有用的检查方法

一旦怀疑环境有问题，先执行：

```bash
python -c "import sys; print(sys.executable)"
```

这条命令会打印出当前正在使用的 Python 程序的完整路径。

后面你创建并激活 `nndl` 环境以后，再运行这条命令，就会看到路径已经变成 `nndl` 环境里的 Python。

这就是为什么“激活环境”会影响后续所有命令。

## 八、Python 环境、虚拟环境和 Conda

### 1. 什么是虚拟环境

**虚拟环境（Virtual Environment）** 可以理解成：  
给一个项目单独准备一间工具房，里面放这个项目专用的 Python 和专用的包。

> 概念：虚拟环境就是一套单独的 Python 执行环境，它尽量只服务于当前项目，不和别的项目混在一起。

这门课里，我们会创建一个名为 `nndl` 的环境。

### 2. 为什么要用虚拟环境

因为不同项目的要求往往不一样。

例如：

- 项目 A 需要 `Python 3.10`
- 项目 B 可能需要别的 Python 版本
- 实验 1 需要 `torch`、`numpy`、`matplotlib`
- 另一个项目可能还需要完全不同的一组包

如果所有东西都直接装到电脑的全局环境里，时间一长就很容易混乱。

### 3. 不用它会怎样

如果不使用虚拟环境，常见问题包括：

- 你给一个项目升级了包，另一个项目突然坏了
- 你已经记不清某个包到底装在了哪里
- 电脑里只有一个越来越乱的 Python 环境

这类问题不是深度学习特有的，而是所有编程项目都会遇到的。

### 4. Conda 到底是什么

> 概念：Conda 是一个专门用来创建和切换环境的工具，也能顺手管理一部分软件包。

在这门课里，我们主要让 Conda 做两件事：

1. 创建环境
2. 切换环境

也就是说，这门课的基本策略是：

- 用 **Conda** 管理环境
- 用 **pip** 安装大部分 Python 包

这样分工比较清楚，也比较适合初学者。

### 5. 在 Mac 上，虚拟环境本质上是什么

这个问题很重要。很多同学会把“环境”想得很抽象，其实它没有那么神秘。

以 Mac 为例，一个 Conda 环境本质上就是某个目录下面的一整套 Python 执行环境。  
这个目录里通常会包含：

- Python 可执行文件
- 各种库文件
- 这个环境自己的 `site-packages`

如果你把 Miniconda 安装在自己的用户目录下，常见情况是：

```text
~/miniconda3/envs/nndl/
```

这里的：

- `~/miniconda3/` 是 Miniconda 的安装位置
- `envs/` 是 Conda 存放各个环境的地方
- `nndl/` 是你这门课的环境目录

在有些机器上，你也可能把 Miniconda 安装到别的位置，所以真实路径会跟着变化。但原理是一样的：  
**环境就是一个实实在在存在于磁盘上的目录。**

### 6. 这个环境目录里会放什么

你可以把 `nndl` 环境想成一个小型的、独立的 Python 世界。  
它里面通常会有类似这样的内容：

```text
nndl/
├── bin/
├── lib/
└── ...
```

其中在 macOS / Linux 上，`bin/python` 往往就是这个环境自己的 Python。  
Python 包通常会装到类似下面的位置：

```text
.../envs/nndl/lib/python3.10/site-packages/
```

也就是说，当你在 `nndl` 环境里执行 `python -m pip install numpy`，这些包一般会被装进这个环境自己的目录里，而不是装到整个系统共用的地方。

### 7. 怎么看自己的环境到底放在哪里

最直接的两种方法是：

```bash
conda info --envs
python -c "import sys; print(sys.executable)"
```

前者会列出 Conda 已知的环境及其路径。  
后者会告诉你当前终端正在使用哪个 Python。

更稳定的做法是：

- 每门课一个环境
- 每个项目一个环境
- 先激活环境，再装包，再运行代码

这样你会清楚地知道：你现在做的所有事，都发生在这门课的环境里。

## 九、包、依赖和包管理器：核心理念是不要重复造轮子

### 1. 先说最核心的理念

写程序时，一个极其重要的原则就是：  
**不要重复造轮子。**

如果别人已经把某件事情做好了，并且这个工具稳定、好用、被很多人验证过，那么你直接用它的成果通常是更高效的。

这就是为什么我们要使用各种库。

### 2. 什么是包、库、依赖

你写的程序，往往不会只用到 Python 自带的功能。  
你还会用到很多别人已经写好的工具，比如：

- `torch`：PyTorch，深度学习实验最核心的库
- `torchvision`：一些常用数据集和视觉工具
- `numpy`：数值计算
- `matplotlib`：画图
- `scikit-learn`：常用机器学习工具函数
- `seaborn`：统计图可视化
- `h5py`：读取 `.h5` 数据文件

这些额外需要的软件包，就叫 **依赖（Dependency）**。

> 概念：依赖就是“你的程序要正常运行，必须借助的其他软件包”。

### 3. 为什么这些包有意义

举个最直接的例子：

- 如果没有 `numpy`，你很多数组和矩阵运算都得自己处理。
- 如果没有 `matplotlib`，你得自己从头解决画图问题。
- 如果没有 `torch`，你得自己写自动求导、张量运算、模型训练框架。

这当然不是不可能，但对实验课来说完全没有必要。  
你真正需要学习的是模型、数据、训练流程和实验结果，而不是重写一遍底层工具。

### 4. 什么是包管理器

> 概念：包管理器就是负责安装、升级和管理软件包的工具。

在 Python 世界里，最常见的包管理器是 `pip`。

> 概念：`pip` 是 Python 最常用的包管理器，可以把别人写好的 Python 包安装到你的当前环境里。

### 5. 为什么推荐 `python -m pip`

这门课里，推荐尽量写成：

```bash
python -m pip install 包名
```

而不是只写：

```bash
pip install 包名
```

原因很直接：  
`python -m pip` 明确表示“请用当前这个 Python 去调用 pip”。这样更容易保证包装到你当前正在使用的环境里。

### 6. 一个容易忽略的事实：依赖还会依赖别的依赖

你安装一个包时，它自己可能还依赖别的包。  
这就像你买了一台电脑，不只是拿到一个外壳，里面还有很多配件一起工作。

所以当你执行：

```bash
python -m pip install jupyter
```

通常不会只安装一个东西，而是会一起装下一串相关依赖。这很正常。

### 7. 一个简单例子

实验 1 里，除了 PyTorch 之外，我们还会用到这些包：

```bash
python -m pip install numpy matplotlib scikit-learn seaborn h5py jupyter ipykernel
```

如果你已经激活了 `nndl`，这些包就会被安装到 `nndl` 环境里。

## 十、版本号、环境快照和复现

### 1. 为什么还要管版本

很多软件包都有版本号，比如 `2.6.0`。  
你可以把它粗略理解成：

- 第一位：大版本
- 第二位：小版本
- 第三位：补丁版本

当一套环境终于可以稳定运行时，我们通常会把当前所有包和版本保存下来。  
这份清单可以看作这套环境的“快照”。

### 2. 为什么这件事重要

因为今天能跑，不代表下个月还能跑。  
软件会更新，依赖会变化，电脑也可能会换。

如果你没有保存环境快照，之后重装环境时就很可能出现这种情况：

- 包名看起来一样
- 环境看起来也差不多
- 结果代码行为却变了

### 3. 这门课最实用的做法

当环境确认能跑以后，顺手把它导出来：

```bash
python -m pip freeze > requirements-exp1.txt
```

这条命令会把当前环境中的包和版本记录到 `requirements-exp1.txt`。

以后如果你想在另一台电脑上尽量复现同样的环境，可以在新环境里执行：

```bash
python -m pip install -r requirements-exp1.txt
```

这就像给当前环境拍了一张照片。  
环境快照不是万能的，但它能显著提高你以后复现实验的成功率。

## 十一、实操：把实验 1 的环境配好

前面这些概念，最后都要落回到真正可执行的步骤。下面我们就按实验 1 的需求，把环境完整配起来。

### 1. 安装 Miniconda

Miniconda 是一个轻量级的 Conda 发行版。  
这门课我们用它来创建和管理 `nndl` 环境。

下载地址：<https://docs.conda.io/en/latest/miniconda.html>

请按你的系统选择对应版本：

- **macOS（Apple Silicon，M 系列芯片）**：选 `macOS Apple Silicon`
- **macOS（Intel）**：选 `macOS Intel x86_64`
- **Windows**：选 `Windows 64-bit`
- **Linux**：选对应的 64 位版本

安装完成后，重新打开终端，输入：

```bash
conda --version
```

如果看到版本号，说明安装成功。

Windows 用户如果一开始总是遇到 `conda` 找不到命令，最省事的做法是先打开 **Anaconda Prompt** 再继续。  
macOS 和 Linux 用户如果看不到 `conda`，先重开一个终端窗口再试。

### 2. 安装 VS Code

下载地址：<https://code.visualstudio.com/>

后面我们会用 VS Code 打开整个课程文件夹，并直接运行 `.ipynb` 文件。

### 3. 创建课程环境

我们统一创建一个叫 `nndl` 的环境，并使用 `Python 3.10`：

```bash
conda create -n nndl python=3.10 -y
conda activate nndl
```

激活完成后，建议立刻检查一下：

```bash
python -V
python -c "import sys; print(sys.executable)"
```

如果看到的是 `Python 3.10.x`，并且路径里出现了 `nndl`，说明环境已经切换成功。

> 注意：每次你重新打开一个新的终端窗口，都要重新执行 `conda activate nndl`

### 4. 安装 PyTorch

这一部分要按你的硬件情况选择。

#### 情况 A：Windows / Linux，且有 NVIDIA 显卡

请直接去 PyTorch 官网选择你的系统、Python 版本和 CUDA 版本，然后复制官网给出的安装命令：

<https://pytorch.org/>

这样做最稳，因为不同电脑的 CUDA 情况可能不同。

#### 情况 B：macOS，或只打算使用 CPU

一般可以直接安装：

```bash
python -m pip install torch torchvision
```

### 5. 安装实验 1 的其他依赖

在已经激活 `nndl` 的前提下，继续执行：

```bash
python -m pip install numpy matplotlib scikit-learn seaborn h5py jupyter ipykernel
```

这些包在实验 1 中大致分别承担这些工作：

| 包名 | 作用 |
|---|---|
| `torch` | 张量运算和模型训练 |
| `torchvision` | 常用数据集与视觉工具 |
| `numpy` | 数值计算 |
| `matplotlib` | 画图 |
| `scikit-learn` | 混淆矩阵等工具函数 |
| `seaborn` | 更方便地画统计图 |
| `h5py` | 读取 `.h5` 文件 |
| `jupyter` | 运行 Notebook |
| `ipykernel` | 让 VS Code 识别当前环境 |

### 6. 把环境注册给 Jupyter / VS Code

执行：

```bash
python -m ipykernel install --user --name nndl --display-name "Python (nndl)"
```

做完这一步以后，VS Code 里的 Notebook 内核列表里通常就能看到 `Python (nndl)`。

### 7. 做一次最小验证

在终端里输入：

```bash
python
```

然后在 Python 里逐行执行：

```python
import torch
import numpy
import matplotlib
import sklearn
import seaborn
import h5py

device = (
    "cuda" if torch.cuda.is_available()
    else "mps" if torch.backends.mps.is_available()
    else "cpu"
)

print(torch.__version__)
print(device)
```

如果没有报错，并且最后打印出了 PyTorch 版本和设备名称，就说明环境基本可用。

最后输入：

```python
exit()
```

退出 Python。

## 十五、常见问题

### 1. 输入 `conda activate nndl` 报错

先完全关闭终端，再重新打开一次。  
如果还不行：

- Windows：优先试 `Anaconda Prompt`
- macOS / Linux：先执行 `conda --version`，确认 `conda` 本身是否正常

### 2. 明明装过包，运行时却说 `ModuleNotFoundError`

最常见的原因有两个：

1. 你没有激活 `nndl`
2. 你在 VS Code 里没有选 `Python (nndl)` 这个内核

先检查这两件事。

### 3. VS Code 里找不到 `Python (nndl)`

重新执行一次：

```bash
python -m ipykernel install --user --name nndl --display-name "Python (nndl)"
```

然后重启 VS Code。

### 4. `code .` 不能用

这不影响你做实验。  
直接在 VS Code 里用 `File -> Open Folder` 打开课程文件夹即可。

### 5. Apple Silicon 的 Mac 上不能用 `cuda`

这是正常的。  
Apple Silicon 通常使用的是 `mps`，不是 NVIDIA 的 `cuda`。

### 6. 环境越来越乱，已经不知道哪里错了

对初学者来说，很多时候重建环境比硬修更省时间：

```bash
conda remove -n nndl --all
conda create -n nndl python=3.10 -y
conda activate nndl
```

虚拟环境的一个重要好处，就是你可以相对放心地删掉重来。

## 十六、最后总结

你现在应该把下面这条主线记清楚：

1. 终端是你输入命令、查看结果的地方。
2. Shell 是在终端里真正执行命令的程序。
3. UI 是按钮、菜单、图标这类图形界面。
4. 一条命令通常由命令、选项和 argument 组成。
5. `pwd`、`ls`、`cd` 帮你管理当前目录和路径。
6. `.` 表示当前目录，`..` 表示上一级目录。
7. `PATH` 决定电脑会优先找到哪个程序。
8. Conda 用来创建和切换虚拟环境。
9. 虚拟环境本质上是磁盘上的一套独立 Python 目录。
10. Python 包和依赖的核心思想是：不要重复造轮子。
11. `python -m pip install ...` 用来把依赖装到当前环境里。
12. VS Code 里选对 `Python (nndl)`，Notebook 才会真正使用你刚刚配好的环境。

如果你把这条主线真正理顺了，这门课后面的实验，以及以后更复杂的科研项目和 AI 应用搭建流程，基本都还是同一套方法。
