# 实验 1 Git 使用指南

这份讲义的目标很明确：帮助你在实验课里真正学会 Git 最核心的工作方式，而不是只会背几条命令。学完以后，你应该能独立完成下面这些事：

1. 知道为什么课程里要专门讲 Git。
2. 知道 Git 最重要的几个底层概念到底是什么意思。
3. 知道如何在本地记录实验代码的历史，并把它同步到 GitHub。
4. 知道当代码改崩、想试新思路、或者和别人协作时，Git 到底能帮你解决什么问题。

这份讲义不会把 Git 的全部功能一口气讲完。像 `rebase`、`cherry-pick` 这类更进阶的话题，我们这次先不展开。我们只抓住最核心、最常用、最适合实验课的一部分。

---

## 一、为什么实验课要学 Git

很多同学刚开始做实验时，保存代码的方式是这样的：

- `实验1.py`
- `实验1_改好了.py`
- `实验1_最终版.py`
- `实验1_最终版_真的最终版.py`

这样做短时间内看上去还能凑合，但很快就会遇到几个问题：

1. 你记不清每个文件到底改了什么。
2. 你不知道哪一个版本才是真的能跑通的版本。
3. 你想退回昨天的状态时，只能靠猜。
4. 你和同学协作时，很容易互相覆盖文件。

这就是为什么课程里要讲 **版本控制**。

> 概念：版本控制就是系统地记录文件历史，让你知道“什么时候改了什么”，也能在需要时回到过去的某个版本。

> 概念：Git 是最常用的版本控制工具之一，你可以先把它理解成“专门给项目保存历史、管理实验分支和支持协作的时间机器”。

做深度学习实验时，你很容易遇到下面这些真实场景：

- 你今天把训练循环改坏了，想回到昨天还能跑通的版本。
- 你想尝试一个新模型结构，但又不想把现在的主代码弄乱。
- 你在实验室和宿舍两台电脑之间切换，希望代码保持同步。
- 你让 AI 帮你修改了一段代码，想清楚记录“它到底改了什么”。

这些事情，Git 都是为你准备好的。

## 二、先把 Git 看成项目的时间机器

很多初学者会把 Git 理解成“记录删了哪一行、加了哪一行的工具”。这样理解不算错，但更稳定的理解是：**Git 会把你的项目在某个时刻的状态保存成一张快照。**

> 概念：快照（Snapshot）可以先理解成“给当前整个项目拍的一张全景照片”。

> 概念：提交（Commit）就是把这张快照正式存进 Git 历史里，并附上一句说明。

### 1. 它到底记录了什么

假设你现在在做实验 1，项目里有这些文件：

- `train.py`
- `model.py`
- `README.md`

你先完成了最基础的线性回归训练代码，然后做了一次提交。  
这时 Git 记录下来的，不只是“你改了某几行”，而是“此刻整个项目长什么样”。

后来你又加了损失曲线可视化，再做一次提交。  
这时 Git 又保存了一张新的项目快照。

所以你可以把 Git 历史理解成一条时间线。时间线上的每一个点，都是一次提交留下来的完整项目状态。

### 2. 为什么这很重要

如果你只靠手动复制文件夹来备份，最大的问题是历史很模糊。  
你可能知道“昨天大概改过模型”，但很难精确知道“到底是哪一次修改引入了错误”。

而 Git 的思路很清楚：

1. 每次做出一个相对完整的小改动，就保存一次。
2. 每次保存都留下说明。
3. 以后任何时候都能回头看那次保存到底是什么样。

这会让你的实验历史从“模糊记忆”变成“可检查、可回退、可比较的记录”。

### 3. 每次提交都有自己的身份证号

Git 里的每一次提交，都会对应一个很长的唯一编号。专业上常把它叫作 **哈希值（Hash）** 或 **SHA-1**。

> 概念：哈希值可以先理解成某次提交的身份证号，用来唯一标记这张历史快照。

你不需要一开始就背这个名词，但要知道：  
Git 之所以能非常准确地定位历史，就是因为每次提交都有自己的唯一身份。

当你执行：

```bash
git log --oneline
```

你通常会看到类似这样的输出：

```text
7f3a2b1 完成线性回归训练循环
1c9d8e4 创建实验1项目
```

这里前面那一小串字符，就是每次提交身份证号的缩写形式。

### 4. 历史之间是怎么连起来的

Git 里的提交并不是互相孤立的。  
新的提交会记住“我是从哪一个旧提交接着来的”。

你可以把它想成这样：

```text
A <- B <- C
```

这表示：

- `A` 是更早的提交，
- `B` 是在 `A` 之后产生的，
- `C` 是目前更新的提交。

专业上，这类历史结构常被叫作 **DAG**。你现在不需要记住这个缩写，只要知道一件事：  
**Git 的历史是一条有来源、有顺序、能沿着箭头一路回到过去的结构。**

### 5. 一个最小实验：给项目做第一次提交

如果你在一个新项目目录里第一次使用 Git，可以先做下面几步。

先初始化仓库：

```bash
git init
```

> 概念：仓库（Repository）就是 Git 用来保存项目历史记录的地方。

这条命令的意思是：从现在开始，让 Git 监控这个文件夹，并在里面建立保存历史所需的数据。

然后随时查看当前状态：

```bash
git status
```

这条命令非常重要。  
如果你以后只记住一条最常用的 Git 命令，那往往就是它。因为 Git 很多时候不会主动替你做判断，而是让你先看清当前状态。

如果这是你第一次在这台电脑上使用 Git，记得先到第八部分配置名字和邮箱。然后你就可以把当前文件加入暂存区，并做出第一次提交：

```bash
git add .
git commit -m "创建实验1项目"
```

这里用 `git add .` 是因为“初始化项目”这一步通常本来就希望把当前目录下的文件一起纳入第一次提交。  
提交完成后，再执行一次：

```bash
git log --oneline
```

你就能看到自己刚刚写下的第一条历史记录。

## 三、为什么 Git 中间要多一个“暂存区”

如果 Git 只是“工作区改完就直接保存到历史”，那它的确会简单一点。  
但这样做有一个很大问题：你很容易把逻辑上不应该放在一起的修改，全都混进同一次提交里。

这就是 **暂存区（Staging Area）** 存在的原因。

> 概念：工作区（Working Directory）就是你此刻在电脑上真正看到、真正修改的那些文件。

> 概念：暂存区（Staging Area）就是提交前的一个缓冲区，你可以先挑出“这次准备正式保存”的修改。

> 概念：本地仓库（Local Repository）就是当前这台电脑上真正保存 Git 历史记录的地方。

### 1. 先把三个区域想成三个位置

你可以这样理解 Git 最重要的三个区域：

1. 工作区：你的工作台，代码就在这里改。
2. 暂存区：你的打包箱，准备把哪些东西放进这次提交，先在这里挑好。
3. 本地仓库：你的档案室，真正已经保存下来的历史都在这里。

最常见的流程是：

```text
工作区修改 -> git add -> 暂存区 -> git commit -> 本地仓库
```

### 2. 为什么不直接“全盘保存”

假设你今天同时做了两件事：

1. 修复 `train.py` 里的一个小 Bug。
2. 在 `report.md` 里随手记了几句还没写完的实验笔记。

如果这时候 Git 每次都直接全盘保存，你就会把“已经做完的 Bug 修复”和“还没写完的笔记草稿”混成一个提交。

这样以后再看历史，就会非常乱。

暂存区的作用，就是让你先想一想：  
**这一次提交，到底想正式记录什么？**

### 3. 一个低认知负担的例子

假设你只想把 `train.py` 的修复保存下来，而 `report.md` 还不想提交。

你可以这样做：

先看状态：

```bash
git status
```

再只把 `train.py` 放进暂存区：

```bash
git add train.py
```

然后提交：

```bash
git commit -m "修复线性回归训练循环中的维度错误"
```

这时：

- `train.py` 的修改已经进入历史。
- `report.md` 的草稿还留在工作区。

这正是暂存区最核心的价值：  
你可以有选择地打包，而不是每次都整桌一起塞进箱子里。

### 4. `git commit -m` 每一部分在干什么

上面这条命令可以拆成三部分：

```bash
git commit -m "修复线性回归训练循环中的维度错误"
```

- `git`：表示这是 Git 的命令。
- `commit`：表示把暂存区里的内容正式存进历史。
- `-m`：表示后面要跟一条文字说明，`m` 来自 `message`。

后面的引号内容，就是这次提交的说明。

对初学者来说，一个很好的习惯是：  
**提交说明写“做了什么”，不要只写“改了一下”。**

例如：

- `完成线性回归训练循环`
- `补充数据加载代码`
- `修复 loss 计算中的维度问题`

这些说明以后回头看时会很有用。

### 5. `git add` 成功时经常很安静

这里顺便补一条很容易让新手困惑的事实：  
很多 Git 命令在成功时不会专门打印“成功”。

比如：

```bash
git add train.py
```

这条命令很多时候不会输出任何内容。  
这并不表示它失败了。Git 和终端工具一样，常常在成功时保持安静。更稳定的做法是再执行一次：

```bash
git status
```

看看文件是不是已经进入暂存区。

### 6. 两条最值得常用的查看命令

在真正提交之前，你很值得养成两个习惯。

第一条是查看还没暂存的修改：

```bash
git diff
```

第二条是查看已经暂存、准备提交的修改：

```bash
git diff --staged
```

这两条命令的意义很简单：  
不要凭感觉提交，要先看看自己到底准备交什么进去。

### 7. 关于 `git add .`

很多教程会很快教你：

```bash
git add .
```

这条命令的意思是：把当前目录里几乎所有变化都加入暂存区。

它当然很方便，但对初学者来说，更稳定的做法通常是先学会：

```bash
git add 文件名
```

当你已经明确知道“这次所有修改都属于同一个逻辑步骤”时，再使用 `git add .` 会更合适。

## 四、分支到底是什么：一张会移动的便利贴

很多同学第一次听到“分支（Branch）”，会以为这表示“复制一整份文件夹，再去另一个地方改”。  
这样理解会带来很多误会。

更稳定的理解是：**分支更像一张贴在某次提交上的便利贴。**

> 概念：分支（Branch）可以先理解成“贴在某次提交上的标签”，它告诉 Git：“当前这条开发线进行到了哪里。”

> 概念：HEAD 可以理解成“你现在人正站在哪里”的标记。

### 1. 为什么分支这么重要

做实验时，你经常会遇到这种情况：

- 当前主线代码已经能跑。
- 你想试一个新想法，比如把线性模型换成一个更复杂的网络。
- 你不确定这个想法能不能成功。

这时候如果你直接在主线上大改，一旦失败，原来那份能跑的代码也被你搅乱了。

分支就是为这种场景准备的。

你可以先在主线保持一个稳定版本，然后另外开一条支线去试新想法。  
如果实验成功，就把结果合回来。  
如果实验失败，这条支线删掉就行，主线保持不动。

### 2. 一个简单的画面

假设当前历史是这样：

```text
A <- B <- C   main
               ^
             HEAD
```

这表示：

- `main` 这张便利贴现在贴在提交 `C` 上。
- `HEAD` 说明你当前正站在 `main` 这条分支上。

如果你现在创建一条新分支 `experiment-mlp`，刚创建出来时，画面大致会变成这样：

```text
A <- B <- C   main
               \
                experiment-mlp
```

注意，这并不是复制了一整套历史。  
更直接的理解是：Git 只是又多贴了一张标签。

### 3. 如何创建并切换到新分支

现在更推荐使用：

```bash
git switch -c experiment-mlp
```

这条命令可以拆成这样理解：

- `switch`：切换到别的分支。
- `-c`：如果这个分支还不存在，就顺便创建它。
- `experiment-mlp`：新分支的名字。

比如你在这条分支上做了几次提交后，历史可能变成：

```text
A <- B <- C   main
          \
           D <- E   experiment-mlp
                    ^
                  HEAD
```

这表示：

- `main` 还停留在旧的稳定位置。
- `experiment-mlp` 已经沿着新方向继续往前走了。

### 4. 切回主线时会发生什么

如果你执行：

```bash
git switch main
```

你会回到主分支对应的项目状态。  
刚才在实验分支上做的新修改，如果还没有合并到 `main`，在主线里就不会出现。

这就是分支最实用的地方：  
它让你可以把“稳定版本”和“试验版本”分开。

### 5. 什么时候合并

如果你觉得实验分支已经成功，可以切回主分支再执行：

```bash
git merge experiment-mlp
```

> 概念：合并（Merge）就是把另一条分支上的历史接到当前分支上。

对实验课来说，你现在只需要先记住这个直觉：  
**分支是拿来安全试错的，合并是拿来把试错成功的结果带回主线的。**

### 6. 一个常见提醒：Detached HEAD

Git 里还有一个初学者偶尔会撞到的状态，叫 **Detached HEAD**，中文常译作“游离头指针”。

> 概念：Detached HEAD 可以先理解成“你现在直接站在某次旧提交上，而不是站在一条正常分支上”。

这时你仍然可以查看代码，甚至也能继续改。  
但如果你在这个状态下做了新提交，又没有另外创建分支，那些新历史之后就很容易不见。

所以你现在只要记住一句提醒就够了：  
**如果 Git 提示你进入了 Detached HEAD 状态，先别慌，优先考虑切回正常分支，或者先新建一个分支再继续改。**

## 五、远程仓库、GitHub 和“多台电脑上的同一个项目”

到现在为止，我们讲的都是你这台电脑上的 Git 历史。  
但在真实使用中，代码通常还需要放到云端，以便备份、同步和协作。

这就会遇到 **远程仓库（Remote Repository）** 和 **GitHub**。

> 概念：远程仓库就是放在另一台电脑或云端服务器上的 Git 仓库。

> 概念：GitHub 是最常用的 Git 代码托管平台之一，你可以先把它理解成“专门帮你保存和分享 Git 仓库的网站”。

### 1. 为什么需要远程仓库

如果没有远程仓库，你可能会遇到这些问题：

- 电脑坏了，本地代码也一起没了。
- 在实验室和宿舍两台电脑之间切换时，不知道怎么同步。
- 和同学合作时，只能来回发压缩包。

而有了 GitHub 这类平台后，你就可以：

1. 把本地历史备份到云端。
2. 在不同电脑之间同步进度。
3. 更稳定地和别人协作。

### 2. Git 是分布式的，GitHub 不是“唯一真本”

这里有一个很重要的观念：  
Git 本身是 **分布式版本控制**。

这句话的意思是：你电脑上的本地仓库，本身就已经是一份完整历史。  
GitHub 只是另外一份远程副本，不是你唯一能工作的地方。

这也是为什么很多 Git 命令在离线时照样能用。  
比如看历史、切分支、做提交，这些本地操作都不依赖网络。

### 3. `clone` 和 `init` 适合不同起点

如果远程仓库已经存在，你想把它完整拿到本地，最常用的是：

```bash
git clone https://github.com/username/repository.git
```

> 概念：`clone` 就是“把远程仓库完整复制到本地，并自动建立连接”。

如果项目一开始是在你电脑上创建的，而远程仓库还没连上，那么常见做法是：

先在本地初始化：

```bash
git init
```

然后把远程仓库地址加进来：

```bash
git remote add origin https://github.com/username/repository.git
```

这里的 `origin` 只是一个约定俗成的名字，通常用来表示“默认远程仓库”。

### 4. `push` 和 `pull` 可以先这样理解

把本地提交发到远程，常用：

```bash
git push -u origin main
```

这条命令里：

- `push` 表示把本地提交推到远程。
- `origin` 表示目标远程仓库。
- `main` 表示目标分支。
- `-u` 表示顺手建立追踪关系，以后可以少写一些参数。

把远程最新提交拿回本地，常用：

```bash
git pull origin main
```

这表示：把 `origin` 上 `main` 分支的最新变化取回来，并合并到你当前分支。

### 5. 一个非常贴近课程的场景

比如你在实验室电脑上完成了今天的实验进度，并执行了：

```bash
git push
```

晚上回到宿舍，你在另一台电脑上打开同一个仓库，再执行：

```bash
git pull
```

你就能拿到下午那台电脑上的最新历史。  
这比自己手动拷贝文件夹稳定得多，也更不容易漏文件。

## 六、实验课里最推荐的 Git 工作流

下面这套流程，基本可以覆盖你在课程实验中的大多数情况。

### 1. 开始做实验前，先看状态

进入项目目录后，第一步先执行：

```bash
git status
```

这样做的好处是，你会先知道：

- 当前在哪个分支上。
- 有没有还没提交的修改。
- 有没有文件已经在暂存区里。

这一步会明显降低“我其实带着昨天没处理完的改动继续写了今天实验”的风险。

### 2. 每完成一个清楚的小步骤，就做一次提交

对实验课来说，比较理想的提交粒度通常是：

- 完成数据读取。
- 完成训练循环。
- 补上可视化。
- 修复一个明确的 Bug。

不太建议的做法是：  
连着改两小时，最后把几十个逻辑混在一个超大的提交里。

更稳定的工作流是：

1. 做完一个清楚的小步骤。
2. 先用 `git diff` 看改了什么。
3. 用 `git add` 选择要提交的文件。
4. 用 `git commit -m "..."` 留下清楚说明。

### 3. 风险较高的尝试，先开分支

如果你要尝试一个比较激进的新思路，比如：

- 换掉模型结构，
- 重写训练脚本，
- 让 AI 对多个文件做较大修改，

更稳妥的做法是先开一条分支：

```bash
git switch -c experiment-new-model
```

这样做的好处是，你可以放心试，不会直接影响主线。

### 4. 每次让 AI 改代码前后，都看一眼 Git

这是现在尤其值得强调的一条习惯。

在让 AI 修改代码之前，你可以先执行：

```bash
git status
git diff
```

改完之后再执行一次：

```bash
git diff
```

这样你就能很清楚地看到：  
AI 到底帮你改了哪些文件、改了哪些地方。

这比单纯在编辑器里看一眼更稳定，因为 Git 是按版本历史的角度在帮你检查变化。

### 5. 一天结束前，把能跑通的状态推到 GitHub

对实验课来说，一个非常好的习惯是：

1. 当天实验结束前，确保至少有一个能跑通的提交。
2. 把它推到远程仓库。

这样即使你第二天改坏了，本地电脑出问题了，或者 AI 帮你生成了一堆你还没完全看懂的修改，你也始终有一个云端可回退的稳定点。

## 七、当代码改崩时，Git 怎么帮你救命

Git 最让人安心的地方，不只是“能保存历史”，而是“历史一旦保存下来，就能真正拿来排查问题”。

### 1. 先学会看历史

查看提交历史最常用的是：

```bash
git log --oneline
```

这条命令会把历史压缩成比较易读的形式，让你快速看到：

- 每次提交的缩写编号，
- 提交说明，
- 大致先后顺序。

很多时候，程序昨天还正常，今天突然不对劲了。  
这时你第一件事不该是慌着乱删代码，而应该先看看：  
**我最近到底做了哪些提交？**

### 2. 为什么“提交说明写清楚”会救命

假设你的历史长这样：

```text
7f3a2b1 修复 loss 计算维度错误
4d8e5c0 加入训练曲线可视化
2a1f9b3 完成线性回归训练循环
0d5c3e1 创建实验1项目
```

即使你现在暂时记不清细节，看到这些说明，你也很容易大概判断：

- 哪一步可能引入了问题，
- 哪一步是稳定版本，
- 接下来该先查哪段代码。

所以好的提交说明不是为了好看，而是为了以后排查问题时能更快定位。

### 3. 一个值得知道的高阶工具：`git bisect`

当历史比较长时，Git 还有一个非常厉害的排查工具，叫 `git bisect`。

> 概念：`git bisect` 可以理解成“帮你在提交历史里做二分查找的工具”，用来定位“到底是哪一次提交把程序弄坏了”。

假设：

- 你知道很早以前某个版本是好的；
- 你知道当前版本是坏的；
- 但中间有很多次提交，不知道哪一次开始出问题。

这时，`git bisect` 的思路是：

1. 先跳到历史中间某个版本；
2. 让你判断这个版本是“好”还是“坏”；
3. 再继续在剩下的一半历史里查。

这就是典型的二分查找思路。

如果中间有 1000 次提交，你不需要一条一条猜。  
理论上只要判断十来次，Git 就能把范围缩得非常小。

### 4. `git bisect` 最基本的使用流程

下面是一个只需要先看懂流程、不需要立刻背下来的版本：

先开始排查：

```bash
git bisect start
```

告诉 Git 当前版本是坏的：

```bash
git bisect bad
```

再告诉 Git 某个旧提交是好的：

```bash
git bisect good 2a1f9b3
```

然后 Git 会自动把你切到中间某次提交。  
你测试一下这个版本：

- 如果这个版本有问题，就执行 `git bisect bad`
- 如果这个版本没问题，就执行 `git bisect good`

最后排查结束后，别忘了回到正常状态：

```bash
git bisect reset
```

你现在不一定立刻会用上它，但我很希望你先知道：  
**Git 的历史记录不是摆设。只要历史足够清楚，Git 甚至能用算法帮你查 Bug。**

## 八、第一次使用前的准备和常用命令

到这里为止，核心思路已经讲完了。最后把最常用、最值得先学会的一组命令收一下。

### 1. 第一次用 Git 前，先配置你的身份

在当前电脑上第一次使用 Git 时，建议先配置名字和邮箱：

```bash
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"
```

这里的 `--global` 表示：对这台电脑上你之后使用的 Git 仓库都生效。

这两条信息会出现在提交记录里。  
如果以后你要和 GitHub 协作，这一步通常也很有必要。

### 2. 一组最值得先记住的命令

下面这几条就是实验课最常用的一组：

```bash
git status
git diff
git add 文件名
git commit -m "说明"
git log --oneline
git switch -c 分支名
git switch main
git push
git pull
```

如果你一开始只把这几条用熟，已经足够完成大多数实验课任务。

### 3. 这些命令分别最适合什么时候用

- `git status`：每次开始前先看状态。
- `git diff`：提交前先看改动。
- `git add 文件名`：有选择地把文件加入暂存区。
- `git commit -m "说明"`：把这次小步骤正式存进历史。
- `git log --oneline`：回头看历史。
- `git switch -c 分支名`：开始一个新尝试。
- `git switch main`：回到主线。
- `git push`：把本地提交同步到远程。
- `git pull`：把远程最新进度拿回本地。

## 九、小结

这份讲义想帮你建立的是一种更稳定的 Git 直觉。

你可以把最核心的内容概括成四句话：

1. Git 会把项目历史保存成一张张可追踪的快照。
2. 暂存区让你能有选择地决定“这次到底提交什么”。
3. 分支让你可以安全试错，不必在主线上硬改。
4. 远程仓库让你能备份、同步，也能和别人协作。

如果再往前走一步，你还会发现：  
只要提交历史够清楚，Git 不只是保存过去，它还能帮你查出“问题到底是从哪一次开始出现的”。

对实验课来说，最值得养成的习惯是这一条：

**每完成一个清楚的小步骤就提交一次，风险较高的尝试先开分支，一天结束前把稳定版本推到 GitHub。**

如果你把这套习惯用熟了，后面的实验、科研代码和借助 AI 搭建应用的过程都会稳很多。
