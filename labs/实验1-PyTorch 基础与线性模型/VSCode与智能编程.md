# 实验 1：VS Code 与智能编程入门

这份讲义的目标很明确：帮助你在实验课里真正用起来现代编辑器和 AI 工具，而不是只知道几个按钮的名字。学完以后，你应该能独立完成下面这些事：

1. 知道为什么课程里要专门讲 VS Code、Trae 和 AI 辅助编程。
2. 知道现代编辑器最核心的几项能力到底解决了什么问题。
3. 知道 Chat 式对话和 Agent 式自动执行分别适合什么场景。
4. 知道在做实验、看报错、改代码和写小应用时，怎样把这些工具用成一个稳定的工作流。

这份讲义不会把所有功能都讲全。我们只抓住最核心、最常用、最适合实验课的一部分。

---

## 一、为什么实验课要先学编辑器和 AI

很多同学刚开始写代码时，会把“写程序”理解成“打开一个文本框，把字符敲进去”。这种理解只对了一半。

做实验时，你真正要面对的事情通常是这些：

- 打开一整个项目目录，而不是只看一个文件。
- 在多个 Python 文件之间来回切换。
- 运行 Notebook、脚本和终端命令。
- 读懂报错，找到出问题的位置。
- 修改代码后再次运行，确认问题有没有真的解决。

如果你只用最简单的文本编辑器，这些事情都会变得很费力。你能写字，但很难高效地读项目、定位问题和验证结果。

这就是为什么课程里要讲现代编辑器。

> 概念：VS Code 是一个写代码、看文件、运行程序和管理项目的编辑器，可以把它理解成程序员常用的工作台。

> 概念：Trae 是在现代编辑器基础上加入更多 AI 能力的开发环境，很多基本操作和 VS Code 很像。

> 概念：IDE 是“集成开发环境（Integrated Development Environment）”的缩写，意思是把写代码、看项目、运行程序、调试和 AI 协作放在同一个地方。

这份讲义同时适用于 VS Code 和 Trae。  
如果你只需要基本的写代码功能，两者的核心操作思路很接近。  
如果你想进一步让 AI 帮你解释代码、修改文件、运行命令，那么 Trae 这类工具会更方便。

## 二、现代编辑器到底替你解决了什么问题

现代编辑器真正强的地方，不是界面更好看，而是它会在你写代码的时候持续帮你“理解项目”。

这背后有一个很重要的概念，叫 **LSP**。

> 概念：LSP 是“语言服务器协议（Language Server Protocol）”的缩写，你可以先把它理解成编辑器背后那个懂 Python 语法和项目结构的助手。

你不需要记住这个缩写的技术细节，但要知道它给你带来了三类非常关键的能力。

### 1. 写错时立刻提醒你

假设你在文件里写了这样一行代码：

```python
imprt math
```

这里把 `import` 拼成了 `imprt`。现代编辑器通常会马上用红色波浪线、提示框或者悬停说明告诉你：这一行有问题。

这一步的意义很大。因为它把“最后运行时才发现错误”，变成了“刚写下去就发现不对劲”。

如果没有这种能力，你往往要等到程序真正执行时才看到报错。那时你已经往后写了很多行代码，回头找问题会更累。

对实验课来说，这种即时提醒非常重要。很多初学者卡住，并不是因为题目太难，而是因为一个拼写错误、一个括号没闭合，或者一个变量名写错了。

### 2. 自动补全，减少重复打字和拼写错误

现代编辑器还会根据你正在写的内容，主动猜测你接下来可能想输入什么。

比如你已经写出：

```python
import torch.nn as nn
```

然后你输入：

```python
nn.Li
```

编辑器很可能会主动提示 `nn.Linear`。  
你只需要按下回车或 `Tab`，就能把它补全。

这样做的好处有两个：

1. 可以明显减少打字量。
2. 可以降低拼写错误的概率。

当你后面开始接触 `DataLoader`、`Linear`、`ReLU`、`optimizer.step()` 这些名字时，这种能力会非常实用。

### 3. 让你快速看懂一个项目

写实验代码时，最常见的困难之一不是“不会写”，而是“看不懂这个项目已经写了什么”。

现代编辑器能帮你做很多快速定位动作，比如：

- 鼠标悬停查看说明。
- 查看某个函数需要哪些参数。
- `Ctrl`/`Cmd` 加点击，直接跳到定义位置。
- 在整个项目里查找某个函数名或变量名。

举个很真实的场景：  
你在 `train.py` 里看到了 `model(x)`，但你不知道模型到底长什么样。  
这时你可以直接跳到模型类定义的位置，看它的 `forward()` 是怎么写的。

如果没有这种能力，你往往只能手动翻文件、全局搜索、来回滚动。项目一大，认知负担就会明显增加。

### 4. 一个很适合课堂演示的小实验

你可以马上做下面三件小事：

1. 故意输入 `imprt math`，看看编辑器会不会提示错误。
2. 输入 `nn.Li`，观察自动补全会不会弹出来。
3. 找到一个你不熟悉的函数或类，试一次 `Ctrl`/`Cmd` 加点击跳转。

如果你亲手做过这三个动作，就会很直观地感受到：现代编辑器并不只是“写代码的地方”，它还是“读懂代码的地方”。

## 三、先把编辑器当成工作台，而不是一张白纸

很多同学第一次打开 VS Code 或 Trae，会把注意力放在“中间那块能打字的区域”上。其实更重要的是，你要把整个编辑器看成一个工作台。

最常用的通常有四个区域：

1. 左边的文件区：看项目目录，切换文件。
2. 中间的编辑区：真正阅读和修改代码。
3. 下方的终端区：运行命令、看输出、看报错。
4. 右边的 AI 对话区：提问、解释、改代码、分析问题。

这里先补一个在第二部分里会反复用到的词。

> 概念：终端是一个让你输入文字命令、查看电脑返回结果的窗口。

如果你已经看过第一部分《环境配置指南》，你应该知道：  
很多实验相关操作，比如装包、运行脚本、检查环境、看报错，本来就离不开终端。  
现在只是把这个终端放进了编辑器里，于是“看代码”和“运行代码”就被放到了同一个工作台上。

### 1. 打开项目时，尽量打开整个文件夹

这一步非常重要。

更稳定的做法是：

1. 选择 `File -> Open Folder`。
2. 打开课程项目的根目录，例如 `nndlClass`。
3. 再从左侧文件树里进入 `labs/实验1-PyTorch 基础与线性模型/`。

这样做的好处是：

- 编辑器能看到整个项目结构。
- 终端通常会从项目目录开始工作。
- AI 也更容易理解“你现在到底在做哪个实验”。

如果你只是双击打开某一个孤立文件，很多上下文都会丢掉。  
例如相对路径、同目录文件、项目结构和相关脚本，编辑器与 AI 都更难一起看懂。

### 2. 第一次进入项目后，先确认这几件事

刚打开实验目录时，你可以先做一个最基本的检查：

1. 左侧能看到本次实验相关的文件和文件夹。
2. 底部终端能正常打开。
3. 如果你在看 Notebook，右上角已经选到了正确的 Kernel。

> 概念：Kernel 可以先理解成 Notebook 背后真正负责运行代码的那套 Python 环境。

这一步看起来很小，但它能帮你提前排掉很多问题。  
有些同学明明代码没错，却一直报奇怪的错误，最后发现只是 Notebook 连错了环境。

## 四、先让 AI 做助手：Chat 与行内补全

现代编辑器里的 AI，最适合初学者入门的用法，不是“一句话把整个项目全做完”，而是先让它做一个愿意解释、愿意示范、也愿意帮你补几行代码的助手。

> 概念：大语言模型（LLM）可以理解成一种能读懂自然语言、也能生成代码和解释文字的 AI 模型。

> 概念：Prompt 就是你给 AI 的任务说明，写得越清楚，AI 越容易帮对忙。

> 概念：行内补全（Inline Completion / Inline Chat）就是 AI 直接在你正在编辑的代码旁边给出补全或修改建议。

### 1. Chat 模式最适合学习和小改动

如果你只是想做下面这些事，那么 Chat 式对话通常已经够用了：

- 解释一段代码在做什么。
- 解释一个报错可能是什么意思。
- 把一段自然语言思路变成一个小函数。
- 给当前函数补注释。
- 按你的要求修改当前文件中的少量内容。

这种模式的优点是节奏慢、控制权强，很适合实验课。

比如你可以直接问：

```text
请用零基础学生能听懂的方式，解释当前文件里这个函数每一行在做什么，不要修改代码。
```

或者：

```text
我现在在做线性回归实验。请只修改当前函数，帮我补全缺失的部分，不要引入新的第三方库。
```

这类提问有一个共同特点：范围清楚，目标清楚，限制也清楚。  
这样 AI 更容易给出你真正能用上的结果。

### 2. 行内补全很适合把一句想法变成几行代码

除了对话框，很多编辑器还会在你写代码时直接给出灰色的建议代码。

比如你在文件里写下一句注释：

```python
# 读取 csv 文件，并画出 loss 曲线
```

接下来，编辑器可能就会在下面生成一段候选代码。  
如果你觉得合理，可以按 `Tab` 采纳；如果觉得不合适，就继续改你的描述，或者自己重写。

这种用法特别适合下面两类场景：

1. 你已经知道自己想做什么，只是懒得把样板代码一行一行敲完。
2. 你知道大方向，但不确定具体 API 怎么写，想先看一个草稿。

它的价值，不只是“省几秒钟打字”，更重要的是帮你把想法更快地变成一个可以运行的初稿。

### 3. 提问时，尽量把四件事说清楚

当你发现 AI 老是答偏时，问题往往不在于它完全不会，而在于你的描述太大、太散，或者缺了关键上下文。

更稳定的 Prompt 结构通常包含四部分：

1. 目标：你想让它做什么。
2. 范围：只看哪个文件、哪个函数、哪一段报错。
3. 限制：不要改哪里，不要引入什么。
4. 输出形式：解释、代码、步骤，还是最小修改方案。

例如：

```text
请阅读当前打开的文件。目标：补全线性回归训练循环。范围：只修改 TODO 标记处。限制：不要改函数签名，不要引入第三方库。输出：先给出思路，再给出代码。
```

再比如：

```text
下面是完整报错。请先解释最可能的原因，再告诉我应该检查哪个文件、哪一行，不要直接大改代码。
```

这类问法特别适合实验课，因为它能把“AI 帮我做事”和“我自己理解过程”放在一起。

## 五、什么时候切到 Agent 或 SOLO 模式

当任务开始变成“不是一句话就能完成的小修改”，而是需要连续执行很多步时，单纯的 Chat 就会慢下来。这时候你可以开始理解另一类工具：Agent。

> 概念：Agent 可以理解成“会自己调用工具做事的 AI 助手”，它不只会回答问题，还能读文件、改文件、运行命令。

> 概念：SOLO 模式是 Trae 里更偏向 Agent 的工作方式，适合把一个目标拆成多步自动推进。

> 概念：Agent Harness 可以先理解成“让 AI 真正能在你的项目里动手操作的那层工具外壳”。

你可以把差别理解成这样：

- Chat 更像你在和一个懂代码的助教对话。
- Agent 更像你给一个会动手的助教下达目标，然后它自己去查文件、改代码、运行命令，再回来汇报。

### 1. 什么时候继续用 Chat

下面这些情况，继续用 Chat 往往更稳：

- 你想先听解释。
- 你只改一个函数或一个文件。
- 你还不确定应该怎么改，想先讨论思路。
- 你担心改动范围太大，想逐步确认。

### 2. 什么时候更适合用 Agent

下面这些情况，Agent 会明显更省力：

- 一个问题涉及多个文件。
- 改完后还要重新运行程序或测试。
- 你希望 AI 帮你完成“读报错 -> 修改 -> 再运行 -> 再修复”的连续流程。
- 你已经把目标说得很清楚，想让它一次推进多步。

一个非常典型的例子是：  
你运行程序后，发现缺少依赖包或者某个导入写错了。  
这时如果只用网页上的普通 AI，你通常要来回复制报错、复制代码、手动运行命令。  
如果用编辑器里的 Agent，它就有机会在同一个环境里连续完成这些动作。

### 3. Trae 里几个常见的 Agent 角色

如果你使用 Trae，最常见的几个名字大致可以这样理解：

- `Chat` 或普通 IDE 对话：适合学习、解释、小改动。
- `SOLO Coder`：适合已有项目中的改代码、修 Bug、重构。
- `SOLO Builder`：更适合快速搭一个网页原型或展示型小应用。
- 自定义智能体：适合给某类长期重复任务设一个固定角色。

你今天不需要把这些模式全部记住。  
对实验课来说，先把下面这一条记住就够了：

**小任务先用 Chat，连续多步任务再考虑 Agent。**

### 4. 初学者的安全习惯

Agent 的能力更强，意味着你更要保留判断权。

更稳妥的做法是：

1. 命令执行先保持手动确认。
2. 看不懂的命令先问它“这条命令每一部分在做什么”。
3. 不要在来历不明的项目里随便让 AI 全自动运行。
4. 改完之后自己重新运行一次，确认结果是真的对了。

这几条习惯在实验课、科研代码和以后做项目时都非常重要。

## 六、为什么 AI 有时会突然变笨：上下文、记忆和规则

很多同学会有一种体验：同一个 AI，刚开始回答得还挺好，聊着聊着就越来越乱。这里面有一个很关键的原因，叫“上下文窗口”。

> 概念：上下文窗口可以理解成 AI 这一次对话里能临时记住的那一小段信息。

如果你给它的信息太杂、太散、太长，它就更容易抓不住重点。  
所以，和 AI 协作时，一个非常重要的习惯是：**少给无关信息，多给关键证据。**

更好的做法通常是：

1. 直接给相关文件，而不是把整个项目一口气都丢给它。
2. 贴完整报错，而不是只说“它报错了”。
3. 告诉它你预期应该出现什么结果。
4. 一次只解决一个明确问题。

### 1. 为什么“完整报错”很重要

比如下面这两种提问，效果通常差很多：

```text
老师，程序报错了，帮我看看。
```

```text
我在运行 train.py 时得到下面这段完整报错。我的目标是完成线性回归训练。请先判断最可能是导入问题、路径问题还是张量维度问题，再告诉我该先检查哪里。
```

第二种问法的信息更完整，所以 AI 更容易给出有用的答案。

### 2. 项目规则最好写清楚

如果一个项目以后会长期和 AI 协作，你还会看到一种做法：在项目里写固定规则，让 AI 每次都先看这些要求。

> 概念：`agents.md` 可以理解成放给 AI 看的“长期规则说明书”，里面写这个项目的固定要求。

例如，你可以在里面写：

- 解释要面向初学者。
- 默认使用 Python。
- 修改代码时优先做最小改动。
- 不要偷偷引入新的第三方依赖。

这类文件对大型项目特别有用。  
不过在本次实验里，你只要先记住这个思路就够了：**规则写得越清楚，AI 越不容易乱跑。**

### 3. 更大的项目里，还会有“平行工作区”

当你以后开始做更复杂的项目时，可能会遇到多个 AI 同时处理不同任务的情况。那时常见的做法是给它们分开工作区，互不干扰。

这个话题和 Git 有关，第三部分会正式展开。你现在先记住一句话：

**一个目标尽量对应一个清楚的会话和一组清楚的文件。**

## 七、实验课里最推荐的工作流

下面这套流程，基本可以覆盖你在实验课中的大多数情况。

### 1. 先打开整个项目，再确认环境

先打开课程项目根目录，然后进入本次实验的文件夹。  
如果你使用 Notebook，再确认右上角的 Kernel 已经连到正确的环境。

这一步的意义是让“项目、终端、环境、AI”四者处在同一个上下文里。

### 2. 先让 AI 帮你理解任务和文件结构

刚开始不要急着让 AI 直接写完整答案。  
更稳妥的做法，是先让它帮你理解当前实验的结构。

例如你可以问：

```text
这是实验 1 的项目。请先告诉我当前目录里每个主要文件大致负责什么，哪些文件最可能需要我修改。不要直接生成完整答案。
```

这样做的好处是，你会先建立对项目的整体感，而不是一开始就把判断权完全交出去。

### 3. 小范围改动，用 Chat 或行内补全

如果你只是补一个函数、改几行训练代码、给某一段加注释，那么 Chat 和行内补全通常已经够用了。

这时很适合使用“目标 + 范围 + 限制”的描述方式。

### 4. 运行程序时，保留完整输出

不管你是运行脚本还是 Notebook，只要出现错误，都尽量把完整报错保留下来。

因为对人和 AI 来说，完整报错都是排查问题最重要的证据之一。  
不要只截最后一句，也不要只说“这里不对”。

### 5. 先让 AI 解释报错，再决定是否让它修改

这一步很关键。

很多同学一看到红字，就立刻让 AI “全部修好”。这样虽然有时也能成功，但学习效果通常不好。

更适合实验课的做法是分两步：

1. 先让 AI 解释这段报错到底在说什么。
2. 再让 AI 给出最小修改方案。

这样做的好处是，你会慢慢学会自己判断：这是环境问题、路径问题、导入问题，还是代码逻辑问题。

### 6. 连续多步任务，再切到 Agent

如果你已经知道问题大致在哪里，而且修复过程需要：

- 阅读多个文件，
- 修改代码，
- 重新运行，
- 再根据新结果继续修，

那么这时候再切到 Agent 或 SOLO Coder，效率会更高。

对于初学者，我更推荐先让 Agent 给出计划，再开始执行。  
这样你还能在真正修改之前看一眼它准备做什么。

### 7. 每次改完，都自己做一次最终确认

最后这一步不能省。

你至少要确认三件事：

1. 代码是否真的运行通过了。
2. 改动是否只发生在应该修改的地方。
3. 结果是否符合实验要求，而不是“看起来没有报错就算结束”。

这一步会决定你是在“借助 AI 学会开发”，还是“把自己变成只会点接受按钮的人”。

## 八、四个非常适合课堂演示的小实验

如果你要在课上快速体验现代编辑器和 AI，下面四个例子都很合适，因为认知负担比较低，而且和后面的实验联系很直接。

### 1. 拼写错误即时提醒

在一个 Python 文件里输入：

```python
imprt math
```

观察编辑器是否立刻提示错误。  
这个例子最适合用来说明“为什么现代编辑器能帮你更早发现问题”。

### 2. 自动补全

输入：

```python
import torch.nn as nn
nn.Li
```

看看是否会出现 `nn.Linear` 的候选项。  
这个例子可以帮助学生建立“自动补全是在减少重复劳动”的直觉。

### 3. 让 AI 解释代码

选中当前文件里一个短函数，然后问：

```text
请用面向零基础学生的方式解释这段代码，每一行在做什么，不要修改代码。
```

这个例子很适合课堂，因为它把 AI 放在“解释和教学”的位置上，而不是直接替学生做题。

### 4. 让 Agent 帮你完成一个连续动作

故意写一段少了导入的代码，例如：

```python
x = np.array([1, 2, 3])
print(x)
```

然后对 Agent 说：

```text
请帮我运行这段代码，定位报错原因，并用最小修改把它修好。
```

这个例子最适合说明：Agent 和普通对话式 AI 的区别，在于它能把多个步骤连起来推进。

## 九、使用 AI 时最值得养成的四个习惯

最后把最实用的四条习惯集中说一下。

### 1. 先解释，再修改

遇到不懂的代码和不懂的报错，先让 AI 解释。  
这样做的好处是，你会慢慢长出自己的判断，而不是完全依赖它替你按按钮。

### 2. 让它做小而清楚的任务

“把整个实验帮我写完”这类指令通常很不稳定。  
“只补全当前函数中的 TODO”“只解释这段报错”“只修改当前文件”这类任务更容易做好。

### 3. 要求最小改动

很多时候你真正需要的，只是修一个导入、补一段循环、改一个维度。  
如果不加限制，AI 很可能顺手把周围一大片都改了。  
所以“请做最小修改”是一个非常值得常用的要求。

### 4. 保留最后的人工检查

AI 很强，但它仍然会误解任务、会猜错意图，也会写出能运行但不符合实验要求的代码。

所以最后一定要自己检查：

- 文件改对了吗？
- 结果和任务要求一致吗？
- 有没有引入你没看懂的新东西？

这一步不是多余动作，而是你真正掌握工具的关键。

## 十、小结

这份讲义想帮你建立的是一种更现代的实验课工作方式。

你可以把它概括成三层：

1. 现代编辑器先帮你更早发现错误、更快看懂项目。
2. Chat 和行内补全帮你解释代码、补小段逻辑、整理思路。
3. Agent 帮你处理需要连续执行多步的任务。

真正重要的，不是记住多少模式名词，而是养成一个稳定习惯：

**打开整个项目，给清楚的上下文，先理解再修改，改完后自己验证。**

如果你把这套习惯用熟了，它不仅能帮助你完成实验，也能帮助你以后读科研代码、排查报错，以及借助 AI 更快地搭建自己的应用。
